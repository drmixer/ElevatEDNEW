# Data-Driven Design

## Learning Goal
- I can explain how engineers use data to define a problem and evaluate solutions.
- I can distinguish between observations, criteria, and evidence from testing.
- I can use test results to support a design recommendation.

## Introduction
Engineering is not just about building something that seems clever. Strong engineering starts with evidence. Engineers define the problem carefully, gather data about what is going wrong, and then use new data to judge whether a design actually improves the situation.

In this lesson, ask:

1. What data helps define the design problem?
2. What counts as evidence that a design worked better?
3. How do engineers use results to decide what to improve next?

## Key Vocabulary
**Criteria**: The features a successful solution must achieve.

**Constraint**: A limit on a design, such as cost, size, time, or materials.

**Prototype**: An early version of a design used for testing.

**Data**: Recorded observations or measurements.

**Evidence**: Data used to support a claim.

**Iteration**: Revising and testing a design again.

## Phenomenon: A Hallway Noise Problem
A school wants to reduce noise in a hallway near classrooms during lunch. Students test the area and collect this information:

- the average noise level near classroom doors is 78 decibels during lunch
- teachers report that instruction in nearby rooms is interrupted
- the hallway is narrow, so large barriers cannot block walking space
- the school can spend only a limited amount on materials

The question is:

How can engineers use data to define the problem and judge whether a solution improves the space?

## How Data Shapes the Design Problem
Engineers use data before building anything.

The hallway example shows several important inputs:

- a measurable problem: high noise level
- a user need: classrooms need fewer interruptions
- physical limits: the hallway must stay passable
- cost limits: the solution must fit the budget

This matters because engineering problems are stronger when the need is clear and measurable. A vague goal such as "make the hallway better" is too broad to test well.

## Testing Produces Evidence
After building prototypes, engineers gather new data to see whether the design meets the criteria.

Possible test data might include:

- decibel levels before and after the change
- whether people can still move easily through the hallway
- whether the materials remain secure after repeated use
- whether the solution stays within budget

One result is not enough by itself. Engineers compare the evidence to the original criteria and constraints.

## Worked Example 1: Defining Clear Criteria
Suppose a student team proposes this criterion:

"The hallway should be quieter."

Why is that too weak?

A stronger version is:

"The average noise level near classroom doors should decrease by at least 10 decibels during lunch while keeping the hallway open for traffic."

This is stronger because it:

- uses a measurable target
- connects to the real user need
- includes a constraint about movement through the space

## Worked Example 2: Choosing the Better Prototype
Two teams test different sound-dampening panels.

Prototype A:

- lowers noise from 78 decibels to 73
- costs less
- stays attached well

Prototype B:

- lowers noise from 78 decibels to 66
- blocks part of the walkway
- costs more than the budget allows

Which prototype better meets the design goal?

A strong claim is:

"Prototype A is the better engineering choice right now because it improves the problem while still meeting the constraints."

Reasoning:

- Prototype B reduces noise more, but it fails key constraints
- engineering success is not only the biggest improvement in one measurement
- the solution must satisfy both criteria and constraints

## Worked Example 3: Using Data to Plan the Next Iteration
Suppose Prototype A works, but teachers still report some distraction.

What should engineers do next?

A strong explanation is:

"Engineers should revise the design and test again, using the current data to target what still needs improvement."

This is how data-driven design works:

- test
- compare results to the goal
- identify what is still weak
- revise and retest

## Common Mistake: Treating Opinions Like Engineering Evidence
Students sometimes say:

"I like Prototype B more, so it is better."

Preference is not enough.

A stronger method is:

1. identify the criteria and constraints
2. compare each design to the measured results
3. choose the option best supported by evidence

## Summary
Data-driven design helps engineers define problems clearly and evaluate solutions fairly.

Key ideas from this lesson:

1. data helps define the need, criteria, and constraints
2. prototype testing produces evidence about how well a solution works
3. strong engineering recommendations are based on measured results, not guesswork alone

When you analyze a design problem, ask what the data shows before the build, what the tests show after the build, and how the evidence supports the next decision.
